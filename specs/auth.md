# Auth Spec

Two access patterns. Every app implements both or one depending on its audience.

---

## Human Access — Cloudflare Access + Google OAuth

All user-facing routes are protected by Cloudflare Access. Cloudflare handles the Google OAuth flow; the app never sees a Google token. After authentication, Cloudflare injects a signed JWT into every request via the `Cf-Access-Jwt-Assertion` header.

The app's only responsibility: verify that JWT on each request.

```python
import jwt
import requests
from functools import lru_cache
from fastapi import Header, HTTPException, Depends

CLOUDFLARE_AUD = os.environ["CLOUDFLARE_AUD"]  # Access Application audience tag

@lru_cache(maxsize=1)
def get_cf_public_keys():
    team = os.environ["CLOUDFLARE_TEAM"]  # e.g. "yourteam"
    url = f"https://{team}.cloudflareaccess.com/cdn-cgi/access/certs"
    return requests.get(url).json()["keys"]

def require_cf_auth(cf_access_jwt_assertion: str | None = Header(default=None)):
    if not cf_access_jwt_assertion:
        raise HTTPException(status_code=401, detail="Missing Cloudflare Access token")
    try:
        jwt.decode(
            cf_access_jwt_assertion,
            options={"verify_signature": True},
            algorithms=["RS256"],
            audience=CLOUDFLARE_AUD,
            jwks=get_cf_public_keys(),
        )
    except Exception:
        raise HTTPException(status_code=403, detail="Invalid Cloudflare Access token")
```

**Environment variables required:**
- `CLOUDFLARE_TEAM` — your Cloudflare Zero Trust team name
- `CLOUDFLARE_AUD` — the audience tag for this specific application (set by Bootstrap at provision time)

**What Bootstrap does at deploy time:**
1. Provisions the Railway app, gets the live URL
2. Calls Cloudflare API to create an Access Application for that URL
3. Writes `CLOUDFLARE_AUD` into the Railway service env vars
4. Returns the Cloudflare-protected URL (not the raw Railway URL)

Users hit the Cloudflare URL, authenticate with Google, and are proxied through. No Google Console involvement beyond the one-time IdP setup in Cloudflare Zero Trust.

## Machine-to-Machine — API Token

Agent-to-service and service-to-service calls use a shared secret in the `X-App-Secret` header.

```python
APP_SECRET = os.environ["APP_SECRET"]

def require_app_secret(x_app_secret: str | None = Header(default=None)):
    if not APP_SECRET:
        raise HTTPException(status_code=500, detail="APP_SECRET not configured")
    if x_app_secret != APP_SECRET:
        raise HTTPException(status_code=401, detail="Unauthorized")
```

`APP_SECRET` is generated by Bootstrap at provision time and injected as a Railway env var. The calling agent receives it in the Bootstrap response.

---

## Route Classification

| Route type | Auth |
|---|---|
| User-facing (browser) | `require_cf_auth` |
| Agent/machine calls | `require_app_secret` |
| Health check (`/healthz`) | None |

## User Onboarding

This is the standard pattern for granting and revoking access to any app. There are no invite flows, no sign-up forms, no admin panels.

**Granting access:** add an email to `access.yaml` in the app's repo and push to `main`. The GitHub Actions workflow fires, calls the Cloudflare API, and the user can log in within seconds.

**Revoking access:** remove the email from `access.yaml` and push. Same flow.

```yaml
# access.yaml
allowed_emails:
  - you@gmail.com
  - collaborator@gmail.com
```

Bootstrap wires this up automatically at provision time:
1. Writes `access.yaml` with the initial `allowed_emails` from the `create_app` request
2. Creates the Cloudflare Access Application + allow-list policy via API
3. Adds `.github/workflows/sync-access.yml` to the repo (from the repo template)
4. Sets `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_APP_ID`, and `CLOUDFLARE_POLICY_ID` as GitHub repo secrets

After that, the app manages its own access list. Bootstrap is not involved in subsequent changes.

---

## What This System Does Not Use

- Passwords
- Sessions managed by the app
- JWTs issued by the app
- HTTP Basic Auth
